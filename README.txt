Веб-портал агентства недвижимости "Тёплый город"

1. Назначение проекта
Проект представляет собой учебное веб-приложение агентства недвижимости. Сайт позволяет:
- регистрировать и авторизовывать пользователей;
- просматривать публичный каталог объектов;
- создавать, редактировать и удалять объявления через личный кабинет;
- отправлять заявки на просмотр объектов;
- отправлять обращения через форму обратной связи;
- автоматически формировать CRM-записи, календарные события и SMS/push-уведомления.

Архитектура проекта остаётся простой и понятной:
- интерфейс: `index.html`;
- сервер и API: `server.js`;
- хранилище данных: `db.json`;
- изображения и статические файлы: папка `photo`.

2. Что уже реализовано в проекте

2.1. Базовый функционал
- регистрация пользователя с выбором роли `client` или `agent`;
- авторизация по логину и паролю;
- проверка текущей сессии через `/api/auth/me`;
- публичный каталог объявлений;
- фильтрация каталога по типу, комнатам и цене;
- личный кабинет агента/администратора;
- CRUD-операции над объявлениями через AJAX без перезагрузки страницы;
- загрузка фотографий объекта;
- экспорт списка объявлений в JSON;
- форма обратной связи с сохранением в `db.json`;
- запись на просмотр объекта с сохранением в `db.json`.

2.2. Ранее сделанные доработки, которые уже есть в текущей версии
- роли пользователей: `client`, `agent`, `admin`;
- демонстрационные аккаунты `demo` / `demo123` и `admin` / `admin123`;
- хранение сессий через Bearer-токен;
- CSRF-защита для изменяющих запросов;
- базовая санитаризация и валидация входных данных;
- публичный каталог доступен без входа;
- защищённый кабинет и операции изменения объявлений доступны только после входа;
- адаптивная сетка интерфейса для разных размеров экрана;
- единая структура API в формате `/api/...`.

3. Новые правки, реализованные в этой доработке

3.1. Поведение интерфейса после входа
После успешной регистрации или авторизации сайт теперь корректно показывает, что пользователь уже вошёл в аккаунт:
- скрываются кнопки `Регистрация` и `Авторизация` в верхнем меню;
- на главной странице больше не показываются кнопки `Регистрация` и `Войти`, если пользователь уже авторизован;
- блоки с подсказками для гостя заменяются сообщением о том, что аккаунт уже активен;
- при попытке открыть страницы `#register` или `#login` уже авторизованный пользователь автоматически переводится в `#cabinet`.

3.2. Обновление визуальной части
- слева сверху возле названия сайта добавлена отдельная картинка жёлтого домика;
- логотип подключён как статический файл `photo/yellow-house.svg`.

3.3. Функциональные SMS/push, календарь и CRM-интеграции
Ранее эти пункты были указаны как будущий этап. Теперь они работают внутри проекта на уровне встроенной серверной логики и JSON-хранилища.

Что происходит автоматически:

1. При отправке формы обратной связи:
- создаётся запись в `messages`;
- создаётся CRM-лид в `crmLeads`;
- создаётся push-уведомление для администратора в `notifications`.

2. При отправке заявки на просмотр:
- создаётся запись в `viewings`;
- создаётся событие в `calendarEvents`;
- создаётся CRM-лид в `crmLeads`;
- создаётся SMS-запись для клиента в `notifications`;
- создаётся push-уведомление владельцу объявления в `notifications`.

3. В личном кабинете появились отдельные блоки:
- заявки на просмотр;
- календарь;
- CRM-интеграции;
- SMS / Push.

Эти блоки загружаются через API:
- `GET /api/integrations/summary`

4. Структура данных `db.json`
Теперь используются следующие коллекции:
- `users` — пользователи системы;
- `properties` — объявления недвижимости;
- `messages` — обращения через форму обратной связи;
- `viewings` — заявки на просмотры;
- `notifications` — журнал SMS и push-уведомлений;
- `calendarEvents` — события календаря по просмотрам;
- `crmLeads` — CRM-лиды, созданные по заявкам и обращениям.

5. API проекта

5.1. Авторизация
- `POST /api/auth/register` — регистрация;
- `POST /api/auth/login` — вход;
- `GET /api/auth/me` — информация о текущем пользователе.

5.2. Объявления
- `GET /api/properties/public` — публичный каталог;
- `GET /api/properties` — список объявлений текущего пользователя;
- `POST /api/properties` — создание объявления;
- `PUT /api/properties/:id` — редактирование объявления;
- `DELETE /api/properties/:id` — удаление объявления.

5.3. Заявки и обратная связь
- `POST /api/feedback` — отправка обращения;
- `POST /api/viewings` — запись на просмотр;
- `GET /api/viewings` — список заявок для агента или администратора.

5.4. Интеграции
- `GET /api/integrations/summary` — сводные данные по заявкам, календарю, CRM и уведомлениям.

5.5. Администрирование
- `GET /api/admin/messages` — просмотр всех сообщений обратной связи для администратора.

6. Нефункциональные требования и их текущее состояние

6.1. Адаптивный интерфейс для ПК/планшетов/смартфонов
Требование выполнено.
В проекте уже используется адаптивная сетка, карточная структура интерфейса и перенос элементов меню и блоков под разные размеры экрана.

6.2. Простая масштабируемая структура API (`/api/...`)
Требование выполнено.
Все серверные операции организованы в едином формате REST-подобных маршрутов `/api/...`, что упрощает дальнейшее расширение.

6.3. Возможность расширения до полноценной БД и CRM-интеграций
Требование выполнено в учебной реализации.
Сейчас данные хранятся в `db.json`, но структура уже разделена по сущностям:
- пользователи;
- объявления;
- заявки;
- сообщения;
- уведомления;
- календарные события;
- CRM-лиды.

За счёт этого проект можно перенести на полноценную БД без переписывания всей логики интерфейса.
CRM, календарь и уведомления уже реализованы функционально внутри проекта и работают через локальные записи и API.

7. Запуск проекта

7.1. Требования
- Node.js версии 16 или выше.

7.2. Запуск через терминал
Из корня проекта выполните одну из команд:
- `node server.js`
- `npm start`

После запуска откройте в браузере:
- `http://localhost:3000`

7.3. Изменение порта при необходимости
PowerShell:
`$env:PORT=8080; node server.js`

Тогда сайт будет доступен по адресу:
- `http://localhost:8080`

8. Тестовые аккаунты

9. Что показать преподавателю или проверяющему

1. Открыть главную страницу как гость и показать публичный каталог.
2. Зарегистрировать нового пользователя или войти под существующим аккаунтом.
3. Показать, что после входа исчезают кнопки повторной регистрации и авторизации.
4. Открыть личный кабинет и создать новое объявление.
5. Показать редактирование и удаление объявления без перезагрузки страницы.
6. Из публичного каталога отправить заявку на просмотр.
7. Показать, что в кабинете появились:
- заявка на просмотр;
- календарное событие;
- CRM-лид;
- SMS/push-записи.
8. Отправить форму обратной связи.
9. Показать, что создались:
- запись в `messages`;
- CRM-лид;
- push-уведомление для администратора.

10. Ограничения учебной версии
- вместо промышленной СУБД используется `db.json`;
- сессии хранятся в памяти процесса Node.js;
- SMS/push, календарь и CRM работают как встроенные локальные интеграции проекта, без подключения внешних сервисов;
- для полноценной продакшн-версии можно позже подключить PostgreSQL/MySQL, внешние почтовые сервисы, SMS-провайдеров, push-платформы и настоящую CRM.
